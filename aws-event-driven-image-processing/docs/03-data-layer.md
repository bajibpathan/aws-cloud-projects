# 💾 Data Layer Design

## Overview

The Data Layer is responsible for storing metadata about every image uploaded to the Amazon S3 bucket.

After Amazon S3 triggers the AWS Lambda function, the Lambda function extracts important information from the uploaded object and stores it in Amazon DynamoDB. This enables the application to persist image metadata for future querying, reporting, and processing instead of relying solely on CloudWatch Logs.

At this stage of the project, the Data Layer records essential information about each uploaded image while maintaining a serverless and highly scalable architecture.

---

# 🎯 Objectives

The Data Layer should:

- Persist image metadata.
- Store one record for each uploaded image.
- Support future querying and reporting.
- Scale automatically without capacity planning.
- Follow AWS security best practices.
- Minimize operational overhead.

# 💡 Design Decision

## Design Decision #3 – Why Amazon DynamoDB?

### Decision

Amazon DynamoDB was selected as the metadata store for this project.

### Why?

The application stores one independent record for each uploaded image.

Each record contains simple metadata such as:

- Image ID
- Bucket Name
- Object Key
- File Size
- Upload Time
- Status
- ETag

The application does not require:

- SQL queries
- Joins
- Relationships
- Complex transactions

Since this is a serverless, event-driven application, DynamoDB provides automatic scaling, low latency, and minimal operational overhead.

### Alternatives Considered

| Service | Reason Not Selected |
|----------|---------------------|
| Amazon RDS | Requires database administration and relational features that the application doesn't need. |
| Amazon Aurora | Optimized for relational workloads but introduces unnecessary complexity for simple metadata storage. |
| Amazon S3 | Suitable for object storage but inefficient for querying image metadata. |
| Amazon DocumentDB | Better suited for JSON document workloads and introduces additional operational overhead. |

### Trade-offs

**Benefits**

- Fully managed service
- Serverless
- Automatic scaling
- Low operational overhead
- Native Lambda integration
- High performance

**Limitations**

- No SQL joins
- Different query model compared to relational databases
- Data model must be designed around access patterns

For this project, these trade-offs are acceptable because the metadata is simple and access patterns are straightforward.

---

# Why Amazon DynamoDB?

Amazon DynamoDB was selected because it is a fully managed, serverless NoSQL database that integrates seamlessly with AWS Lambda.

The project only needs to store and retrieve image metadata without requiring relational queries or complex joins. DynamoDB automatically scales based on workload and eliminates the need to provision or manage database servers.

This makes DynamoDB an ideal choice for event-driven serverless applications.

---

# Why Not Amazon RDS?

Amazon RDS is designed for relational workloads that require:

- SQL queries
- Table joins
- Relationships
- Transactions across multiple tables

For this project:

- Each uploaded image is stored as an independent record.
- No relationships exist between records.
- Query requirements are simple.

Using Amazon RDS would introduce unnecessary database administration and operational overhead.

---

# Implementation Details

## Partition Key

The table uses:

```text
ImageId (String)
```

Each uploaded image receives a unique UUID generated by the Lambda function.

Using a UUID guarantees uniqueness even if multiple users upload files with identical names.

---

## Why Not Object Key as the Partition Key?

Although the S3 Object Key appears unique, users may upload files with identical names or overwrite existing objects when versioning is enabled.

Using a generated ImageId provides a consistent and unique identifier for every metadata record.

---

## Capacity Mode

The table uses:

```text
On-Demand Capacity
```

Image uploads occur unpredictably.

On-Demand Capacity automatically scales with traffic and charges only for actual read and write requests.

This removes the need for capacity planning and aligns well with serverless architectures.

---

## Environment Variables

The Lambda function stores the DynamoDB table name in an environment variable.

```text
TABLE_NAME = ImageMetadata
```

This improves maintainability by avoiding hardcoded resource names in the application code.

Future environments (Development, Test, Production) can reuse the same code with different configuration values.

# Security Decisions

## Least Privilege IAM

The Lambda Execution Role has permission only to:

```text
dynamodb:PutItem
```

on the **ImageMetadata** table.

The function cannot:

- Delete items
- Scan the table
- Update unrelated tables
- Create or delete tables

This follows the Principle of Least Privilege.

---

## Serverless Database

Amazon DynamoDB removes the need to manage:

- Database servers
- Operating systems
- Backups
- Capacity planning

This reduces operational overhead and allows the team to focus on application development.

# Lessons Learned

- Amazon DynamoDB integrates seamlessly with AWS Lambda.
- Serverless databases remove infrastructure management.
- UUIDs provide reliable unique identifiers.

# Challenges

## Challenge

Initially, it was unclear whether Amazon RDS or Amazon DynamoDB would be the better choice.

### Resolution

After evaluating the application's requirements and access patterns, Amazon DynamoDB was selected because it provides a simpler, serverless, and more scalable solution for storing image metadata.
